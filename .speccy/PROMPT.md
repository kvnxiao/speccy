We want speccy to be a spec-driven development engine that can easily be used to take arbitrary greenfield projects from 0 to v1.0 completion, but also be flexible enough to be inserted into an existing brownfield project with little ceremony. We want its engine, the development loop, to be able to power a future multi-agent orchestration harness (e.g. kanban board that operates autonomously, or at the very least, with minimal human interfacing only at the QA steps). The goal is to try to reduce the influence of LLM nondeterminism via providing a robust and deterministic workflow that this CLI tool is meant to provide. Speccy is meant to be lightweight and meant to not enforce a heavy process on HOW someone might go about working on a new project. It tries to provide a defined structure around how AI agents should interact with and implement tasks in any project - by also breaking spec requirements down into smaller, more manageable tasks. This implies a heavy need for pushing to support test driven development within speccy as well - validation is necessary to reduce or outright prevent drift from what a user wants behavior-wise vs. what gets implemented by a non-deterministic LLM agent (and this can add up over time), but we don't necessarily want to make this a hard requirement as not every engineer approaches projects via TDD.

Here are some important things to note about spec driven development kits that skeptics point out:
- "Probably unpopular opinion but SDD is vibe coding with extra steps. While it gives more context and supposedly constraints, it doesn't give better signals to the LLM. I believe SDD is only good for bootstrapping green-field projects, at most. And you really need to "waterfall" plan almost the whole vision."
-  "Spec driven dev makes two assumptions both of which are only partially true:
   - LLMs are good at following instructions
   - It is possible to fully specify what the end result should be before you start coding.
   That last one in software engineering is called "Design Specification" and "Requirements Engineering". Neither is easy to do well and neither gives you a perfect spec."

It would be good to ensure that speccy isn't over-the-top ceremonial baggage that adds more unnecessary friction to the development process.
