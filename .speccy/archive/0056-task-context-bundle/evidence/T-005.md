# Evidence for SPEC-0056 T-005

Red-then-green paper trail for extending the `speccy context` bundle
with three navigation aids (REQ-005): a sibling-task index
(id/state/covers only, no bodies), repo-relative paths to SPEC.md /
TASKS.md / the task's journal file, and a suggested merge-base diff
command computed from git state. "Red" is established by the absence
of the `siblings`, `paths`, and `diff_command` fields on the
`ContextBundle` envelope before this task — T-004 left the envelope
ending at `journal`, so the assertions below were unreachable (the
JSON carried no `siblings` / `paths` / `diff_command` keys, and the
six-task fixture `tasks_md_six` and the net-new
`git::suggested_diff_command` machinery did not exist). Tests live in
`speccy-cli/tests/context.rs`; the fallback unit test lives in
`speccy-cli/src/git.rs`.

## Session 2026-06-10-T005-attempt-1

### Scenario 1 — Sibling index carries id/state/covers only, no bodies (CHK-008)

<red>
At HEAD before this task, `ContextBundle` had no `siblings` field.
`serde_json::Value::get("siblings")` returned `None`, so
`bundle_sibling_index_carries_id_state_covers_only_no_bodies` panicked
at `.expect("bundle has a siblings array")`. There was no sibling
projection at all and no six-task fixture to project from.
</red>

<green>
`bundle_sibling_index_carries_id_state_covers_only_no_bodies` writes a
six-task spec (`tasks_md_six`) whose every task body carries a distinct
`SIBLING_BODY_MARKER_N` string, emits the bundle for T-003, and
asserts:

- the `siblings` array is exactly `[T-001, T-002, T-004, T-005, T-006]`
  in TASKS.md declared order — the selected T-003 is excluded.
- T-001's entry has `state == "completed"` and `covers == ["REQ-001"]`,
  proving state/covers projection (the fixture gives each task a
  distinct state).
- every sibling object's key set is exactly `{covers, id, state}` — no
  `body` field leaks in.
- none of the five siblings' `SIBLING_BODY_MARKER_N` strings appear
  anywhere in the payload, while the selected task's own
  `SIBLING_BODY_MARKER_3` *is* present (it is the `task` entry per
  REQ-003) — proving the absence is sibling-scoped, not a vacuous
  all-absent check. Test command:
  `cargo test -p speccy-cli --test context`.
</green>

### Scenario 2 — Paths resolve from repo root; diff command is merge-base form (REQ-005 behavior)

<red>
Before this task, the envelope had no `paths` or `diff_command`
fields. `bundle_carries_repo_relative_paths_and_diff_command` and
`diff_command_uses_resolved_default_branch_and_runs_from_repo_root`
panicked at `.expect("bundle has a paths section")` /
`.expect("bundle has a diff_command string")`. No default-branch or
merge-base git machinery existed: `git.rs` carried only `repo_sha`,
and the consistency probe correlates by commit-title prefix, never by
merge-base.
</red>

<green>
Two tests, two layers:

- `bundle_carries_repo_relative_paths_and_diff_command` (no git repo):
  asserts the three paths are the repo-relative forward-slash strings
  `.speccy/specs/0042-alpha/{SPEC.md,TASKS.md,journal/T-001.md}`, that
  the SPEC.md and TASKS.md paths resolve to real on-disk files (the
  journal legitimately does not exist for a round-1 task), and that
  outside a git repo the diff command degrades to the runnable
  `git diff main...HEAD` fallback baseline.
- `diff_command_uses_resolved_default_branch_and_runs_from_repo_root`
  (live git repo): stands up a real repo with an initial commit on
  `main`, a simulated `origin/HEAD -> origin/main` remote-tracking
  ref, and a `feature/example` branch with its own commit. Asserts the
  bundle's `diff_command` is `git diff origin/main...HEAD` (the
  resolved default branch in merge-base form, exercising the live
  default-branch probe rather than the fallback), then *runs that exact
  command* from the repo root and asserts it exits 0 and its output
  contains the feature branch's change — proving the suggested command
  is runnable as-is. The test self-skips when git is unavailable; the
  `git.rs` unit test
  `suggested_diff_command_outside_a_repo_falls_back_to_main_baseline`
  covers the no-git fallback path directly.

```
$ git diff origin/main...HEAD   # the bundle's suggested diff_command
diff --git a/change.txt b/change.txt
...
+feature change
(exit 0)
```
</green>
