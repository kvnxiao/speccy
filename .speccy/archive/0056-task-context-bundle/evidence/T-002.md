### Scenario 1 — Ambiguous unqualified selector → shared LookupError class (CHK-001)

<red>
At HEAD before this task, `speccy context` is not a subcommand at
all. The clap enum in `speccy-cli/src/main.rs` had no `Context`
variant, so:

```
$ speccy context T-001 --json
error: unrecognized subcommand 'context'
(clap usage error, exit 2)
```

There was no `report_lookup_error` routing for `context`, so no
ambiguity diagnostic could be produced.
</red>

<green>
With the command wired, an unqualified `T-001` that matches in two
specs surfaces the shared `LookupError::Ambiguous` (library level,
`ambiguous_unqualified_selector_surfaces_lookup_ambiguity`) and, at
the CLI boundary, renders the same diagnostic class `speccy check`
produces via `report_lookup_error`
(`binary_ambiguous_selector_matches_check_diagnostic_class`):

```
$ speccy context T-001 --json    # workspace with SPEC-0042 + SPEC-0043, both have T-001
speccy context: T-001 is ambiguous; matches in 2 specs.
Disambiguate with one of:
  speccy context SPEC-0042/T-001
  speccy context SPEC-0043/T-001
(exit 1, stdout empty)

$ speccy check T-001             # same workspace
speccy check: T-001 is ambiguous; matches in 2 specs.
...
```

Both commands emit the identical "T-001 is ambiguous; matches in 2
specs." headline and enumerate the candidate specs — the same
diagnostic class. The `invoke_json_err` helper asserts stdout is
empty (REQ-001: selector failures exit non-zero with no partial
output). Test command:
`cargo test -p speccy-cli --test context`.
</green>

### Scenario 2 — Qualified selector → one JSON doc, schema_version first (CHK-002)

<red>
Before this task there was no envelope. `speccy context
SPEC-0042/T-001 --json` was an unrecognized subcommand (clap exit 2,
no JSON on stdout). No struct declared `schema_version` as a context
bundle's first field.
</red>

<green>
`qualified_selector_emits_json_with_schema_version_first` asserts
the stdout (a) parses as exactly one `serde_json::Value` with
`schema_version == 1`, and (b) the raw serialized string starts with
`{"schema_version":1` — confirming it is the first serialized field
(`serde_json::Value` does not preserve key order, so the raw-prefix
assertion is the load-bearing one). The struct field order in
`ContextBundle` (`schema_version` declared first) is the contract.

```
$ speccy context SPEC-0042/T-001 --json
{"schema_version":1,"spec":{"id":"SPEC-0042",...},"intent":{...}}
(exit 0)
```
</green>

### Scenario 3 — Intent present, Summary marker absent (CHK-003)

<red>
No bundle existed, so neither the goals/non-goals/decisions intent
slice nor any exclusion of the Summary narrative could be observed.
</red>

<green>
`bundle_carries_intent_and_excludes_summary_marker` builds a SPEC.md
whose intent surfaces carry distinct marker strings
(`GOALS_MARKER_ALPHA/BETA`, `NONGOALS_MARKER`,
`DECISION_MARKER_ONE/TWO`) and whose Summary paragraph and
user-stories carry their own markers (`SUMMARY_NARRATIVE_MARKER`,
`USER_STORY_MARKER`). The test asserts:

- `spec.id/title/status` equal the frontmatter values.
- `intent.goals` contains both goals markers; `intent.non_goals`
  contains the non-goal marker.
- `intent.decisions` is exactly `[DEC-001, DEC-002]` in declared
  order, each with its body (DEC-001 body contains
  `DECISION_MARKER_ONE`).
- The whole payload does NOT contain `SUMMARY_NARRATIVE_MARKER` or
  `USER_STORY_MARKER` — the Summary narrative and user stories are
  excluded by construction (`build_intent` never reads them).

```
$ speccy context SPEC-0042/T-001 --json | grep -c SUMMARY_NARRATIVE_MARKER
0
```
</green>
