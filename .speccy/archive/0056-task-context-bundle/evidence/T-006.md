## Session 2026-06-10-T006-attempt-1

### Scenario 1 ‚Äî Drifted and undrifted bundles share status, drifts are self-scoped (CHK-009)

<red>
At HEAD before this task, `ContextBundle` had no `consistency` field.
`serde_json::Value::get("consistency")` returned `None`, so
`drifted_and_undrifted_bundles_share_status_but_scope_drifts_to_self`
panicked at `.expect("bundle carries a consistency section")`. There
was no consistency projection at all, no per-task drift filter, and no
multi-task fixture or git-repo helper to produce drift from.
</red>

<green>
`drifted_and_undrifted_bundles_share_status_but_scope_drifts_to_self`
writes a six-task spec where T-001 and T-002 are `completed` while the
rest are `pending`, stands up a real git repo with one initial commit
and no `[SPEC-0042/T-NNN]:` commits (so each `completed` task surfaces a
`state_completed_no_commit` blocking drift), then asserts:

- the bundle for the drifted T-001 carries a non-ok status and a
  `drifts` list of exactly `["T-001"]` ‚Äî T-002's drift, which exists in
  the workspace, does not appear (proving the task-scoped filter, not a
  vacuous all-absent check).
- the bundle for the undrifted T-003 (`pending`) carries the *same*
  non-ok workspace status but an empty `drifts` list ‚Äî proving status
  is workspace-level while drifts are self-scoped.
- both the T-001 and T-003 binary invocations exit 0 ‚Äî `speccy context`
  never refuses on drift (REQ-006 done-when: drift never changes the
  exit code). The library `run` returning `Ok` (asserted via the `?` in
  `invoke_json`) plus the `Command::cargo_bin(...).assert().success()`
  on both selectors establish the exit-0 contract at both layers. Test
  command: `cargo test -p speccy-cli --test context`. Self-skips when
  git is unavailable.
</green>
