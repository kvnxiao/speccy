# Evidence for SPEC-0056 T-007

Executable-contract paper trail for the REQ-007 size invariant: the
`speccy context` bundle for a fixed task is invariant to spec growth
except for one sibling-index line per added task. The test lives in
`speccy-cli/tests/context.rs`
(`bundle_size_scales_with_task_not_spec`). This task adds a
property-style test that pins existing bundle behaviour (the
size-bounding logic landed across T-002..T-006); no production code
changed. "Red" is established below by what the test would catch if any
of the three spec-growth operations leaked into the task-scoped bundle.

## Session 2026-06-10-T007-attempt-1

### Scenario 1 — grow by uncovered req + sibling task + foreign journal round; diff is exactly one sibling entry (CHK-010)

<red>
Before this test existed, no executable contract pinned the size
invariant — REQ-007 was prose intent only. The test is written so it
fails loudly under any regression that lets spec growth leak into the
bundle:

- If `build_requirements` stopped filtering by the task's `covers`,
  the added uncovered `REQ-004` would appear in the `after` bundle's
  `requirements` array, breaking both the `strip_siblings` deep-equality
  assertion and the explicit `!after_str.contains("REQ-004")` guard.
- If the journal section read anything beyond the selected task's own
  `journal/T-001.md`, the foreign `journal/T-002.md` round would flip
  `journal.exists` to `true`, breaking the deep-equality assertion and
  the explicit `journal.exists == false` guard.
- If a sibling entry carried body text (or extra fields), the
  `["covers","id","state"]` key-set assertion would fail.

I confirmed the test is not vacuous by reasoning about each guard
against a concrete mutation it would catch; the deep-equality of the
siblings-stripped payloads is the load-bearing "nothing else changed"
assertion.
</red>

<green>
`bundle_size_scales_with_task_not_spec`:

1. Emits the bundle for `SPEC-0042/T-001` against a spec with three
   requirements and a single task `T-001` covering `REQ-001`
   (`before`); asserts zero siblings.
2. Grows the spec three ways REQ-007 enumerates — adds an uncovered
   `REQ-004`, appends sibling task `T-002` (covering `REQ-002`), and
   writes a one-round `journal/T-002.md` for the foreign task — then
   re-locks `spec_hash_at_generation` to the edited SPEC.md's SHA-256
   (mirroring the reconcile step; the bundle does not read the hash).
3. Re-emits for the same `T-001` (`after`).
4. Normalizes the `consistency` section in both payloads to a fixed
   sentinel (the only field a SPEC edit can perturb, per CHK-010
   "consistency fields normalized").
5. Asserts: `after.siblings.len() == 1`, the one entry is `T-002`
   carrying only `covers`/`id`/`state`, and the siblings-stripped
   payloads are deep-equal — the executable form of "differs by exactly
   one sibling-index entry and nothing else."

Test command: `cargo test -p speccy-cli --test context`.
</green>
