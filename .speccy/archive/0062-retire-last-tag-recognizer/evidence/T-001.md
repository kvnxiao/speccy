## Session 2026-06-12-T001-attempt-1

### Scenario 1 — Frontmatter-present, trailing-corruption journal recovers to the byte past `</implementer>` (CHK-002)

Both preserved fixtures pin the recovery offset to
`find("</implementer>") + "</implementer>".len()` for a journal that is
well-formed through its first block and then corrupt. The new helper homed
in `journal_xml` reuses `scan_tags`; the offset it records for a depth-0
close is `RawTag.span.end` (the byte just past the close tag's `>`), which
equals the value the deleted hand-rolled scan produced.

<red>
Pre-edit (before re-homing the helper / fixing the fixtures), the relocated
unit `last_well_formed_offset_finds_close_of_implementer` and the integration
fixtures failed. The original `consistency.rs` fixtures carried NO
frontmatter and relied on the missing-frontmatter parse failure; under the
new helper (which gates on `split_required`) a frontmatter-less journal
recovers to 0, and a trailing tag that errors `scan_tags` also recovers to 0:

```
$ cargo test -p speccy-core --lib last_well_formed
test ...last_well_formed_offset_finds_close_of_implementer ... FAILED
  left: 0
 right: 153
```

The `0` is the new helper short-circuiting on the frontmatter guard / the
errored scan — proving the helper does derive from `scan_tags` rather than
the old whole-source `find('<')` loop, which ignored frontmatter entirely.
</red>

<green>
Post-edit: each fixture carries minimal valid frontmatter and a trailing
*whitelisted* open tag with no matching close (`<implementer round="2">`),
which fails strict parse (unbalanced element) while `scan_tags` still pairs
the first block. The recovery offset is the byte just past the first
`</implementer>`:

```
$ cargo test -p speccy-core --lib last_well_formed
test ...last_well_formed_offset_finds_close_of_implementer ... ok
test ...last_well_formed_offset_zero_when_no_close_tag ... ok
test ...last_well_formed_offset_zero_when_frontmatter_missing ... ok
test result: ok. 3 passed; 0 failed

$ cargo test -p speccy-core --test consistency_detect detect_journal_xml_malformed
test detect_journal_xml_malformed_is_blocking_with_forward_slash_path ... ok

$ cargo test -p speccy-cli --test consistency journal_xml_malformed
test journal_xml_malformed_reports_kind_path_and_offset ... ok
```

The two integration assertions compute `expected = find("</implementer>") +
"</implementer>".len()` over their own fixture body, so the offset value is
unchanged from today's behaviour — only which code path produces it changed.
</green>

### Scenario 2 — Frontmatter-missing journal recovers to offset 0 (CHK-003)

A journal whose YAML frontmatter is absent makes `split_required` return
`Err` before any tag scan, so the helper short-circuits to 0 — the
legitimate "nothing closed cleanly" value.

<red>
The deleted hand-rolled scan had no frontmatter guard at all: it would have
walked a frontmatter-less source and returned the offset past any close tag
it found, not 0. There was no test pinning the frontmatter-missing → 0
behaviour before this task; CHK-003 is the new guarantee.
</red>

<green>
The relocated unit `last_well_formed_offset_zero_when_frontmatter_missing`
asserts 0 for a frontmatter-less journal carrying a complete
`<implementer>…</implementer>` block:

```
$ cargo test -p speccy-core --lib last_well_formed_offset_zero_when_frontmatter_missing
test ...last_well_formed_offset_zero_when_frontmatter_missing ... ok
test result: ok. 1 passed; 0 failed
```
</green>
