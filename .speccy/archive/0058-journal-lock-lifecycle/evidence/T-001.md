### Scenario 1 — `speccy status --json` stdout stays clean JSON while the subscriber emits a diagnostic on stderr (CHK-008)

The test `status_json_diagnostic_lands_on_stderr_not_stdout` is non-vacuous
because the binary really emits a diagnostic during the command:
`init_tracing()` logs a startup `tracing::trace!("speccy starting", …)` once the
subscriber installs. Run under `RUST_LOG=trace`, that event is live and must
land on stderr; a subscriber miswired to stdout would interleave its formatter
line ahead of the JSON envelope and break the parse. The test asserts both
sides of the contract — the event present on stderr, and stdout still clean
JSON with the event text absent.

<red>
Falsification: rewire the subscriber writer to stdout
(`speccy-cli/src/main.rs:320` `.with_writer(std::io::stdout)` instead of
`std::io::stderr`) and re-run. The startup TRACE no longer reaches stderr (and
its formatter line is pushed onto stdout ahead of the JSON), so the test fails
at the stderr assertion:

```
$ sed -i 's/\.with_writer(std::io::stderr)/.with_writer(std::io::stdout)/' speccy-cli/src/main.rs
$ cargo test -p speccy-cli --test tracing_stdout
running 1 test
test status_json_diagnostic_lands_on_stderr_not_stdout ... FAILED

thread '...' panicked at speccy-cli\tests\tracing_stdout.rs:57:5:
expected the startup TRACE diagnostic on stderr under RUST_LOG=trace; stderr was: ""

test result: FAILED. 0 passed; 1 failed; 0 ignored; 0 measured; 0 filtered out
```

This is the regression CHK-008 exists to guard, and it is a failure that
actually occurs with the corrected test — the writer was then restored to
`std::io::stderr`.
</red>

<green>
With the subscriber correctly wired to `std::io::stderr` in `fn main()` via
`init_tracing()`: driving the real binary with `RUST_LOG=trace speccy status
--json` against a scratch `common::Workspace`, the startup TRACE appears on
stderr, the full stdout payload parses as a single JSON value with
`schema_version == 1`, and the diagnostic text is absent from stdout.

```
$ cargo test -p speccy-cli --test tracing_stdout
running 1 test
test status_json_diagnostic_lands_on_stderr_not_stdout ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

`RUST_LOG=trace` lowers the filter below the startup TRACE so a real diagnostic
is in flight; the clean parse plus the present-on-stderr / absent-from-stdout
assertions prove diagnostics go only to stderr.
</green>
