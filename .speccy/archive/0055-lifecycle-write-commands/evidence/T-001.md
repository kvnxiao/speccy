## Session 2026-06-09-T001-attempt-1

### Scenario 1 — Byte-surgical rewrite touches only the state value (CHK-001)

Unit test
`speccy-core::parse::task_xml::tests::splice_rewrites_only_the_state_value_crlf_multiline_unusual_spacing`.

<red>
Before this task, no `splice_task_state` function existed in
`speccy-core`. The only TASKS.md writer was
`task_xml::render`, which the SPEC explicitly forbids for this rewrite
because it reformats and strips nested blocks. A round-trip through
`render` on the CRLF + multi-line-body + multi-space-attribute fixture
would have re-emitted the file with LF line endings and normalised
attribute spacing — a whole-file diff, not a single-value diff. So the
"differs only in the state value's bytes" assertion was unsatisfiable
with the pre-task code.
</red>

<green>
The fixture is a CRLF TASKS.md with two tasks; task one carries
unusual-but-legal multi-space attribute separation
(`id="T-001"   state="pending"   covers="REQ-001"`) and a multi-line
body that itself contains the literal text `state="completed"` inside
prose. `splice_task_state(src, t1, TaskState::InProgress)` returns a
string byte-equal to `src.replacen("id=\"T-001\"   state=\"pending\"",
"id=\"T-001\"   state=\"in-progress\"", 1)`. The assertions confirm:
the prose-embedded `state="completed"` is untouched (the regex is
scoped to the open-tag span, never the body), the second task's
`state="pending"` is untouched, and re-parsing the rewritten source
yields task one `InProgress` / task two `Pending`. CRLF line endings,
frontmatter, and all whitespace survive verbatim.
</green>

### Scenario 2 — Not-found selector exits non-zero, bytes unchanged (CHK-002)

Integration test
`speccy-cli/tests/transition.rs::not_found_selector_exits_nonzero_and_leaves_bytes_unchanged`,
driving the built `speccy` binary against a scratch workspace.

<red>
Before this task, `speccy task transition` did not exist; clap would
have rejected the `task` subcommand at argument-parse time, so the
"resolves to no task → exit non-zero, file unchanged" behaviour had no
implementation to exercise.
</red>

<green>
A scratch workspace holds `SPEC-0042` with a single `state="pending"`
T-001. Running
`speccy task transition SPEC-0042/T-099 --to completed` (no T-099
exists) exits non-zero with a `not found` diagnostic on stderr, and a
byte read of TASKS.md before and after the invocation is identical.
The resolution failure surfaces from the shared `task_lookup::find`
seam (the same `NotFound` `LookupError` `speccy check` produces),
before any write is attempted.
</green>

### Scenario 3 — All sixteen ordered state pairs classified correctly (CHK-003)

Unit test
`speccy-core::parse::task_xml::tests::classify_covers_all_sixteen_ordered_pairs`,
reinforced by integration tests
`illegal_edge_exits_nonzero_naming_both_states_and_leaves_bytes_unchanged`
and `same_state_is_a_noop_success_leaving_bytes_unchanged`.

<red>
Before this task there was no `classify_transition` function and no
legal-edge table; the state graph lived only in SPEC prose, so "the
six legal edges plus four no-ops succeed and the other six are
illegal" could not be asserted against code.
</red>

<green>
The test iterates all 4×4 ordered `(from, to)` pairs and tallies the
`TransitionKind` returned by `classify_transition`: exactly 6 `Legal`
(matching the closed edge set), exactly 4 `NoOp` (the same-state
diagonal, DEC-003), and exactly 6 `Illegal`, with cross-checks that no
edge is mis-bucketed. The integration tests confirm the CLI behaviour:
`pending → completed` exits non-zero with a stderr diagnostic naming
both `pending` and `completed` and "not in the legal state graph",
leaving TASKS.md byte-identical; `pending → pending` exits 0 leaving
TASKS.md byte-identical.
</green>
