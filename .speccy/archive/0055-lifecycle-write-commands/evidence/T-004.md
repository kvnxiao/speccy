# Evidence for SPEC-0055 T-004

Red-then-green paper trail for routing `journal append` vet block
types to VET.md (REQ-004). The proof lives in the integration suite
at `speccy-cli/tests/journal_append_vet.rs` (drives the built
`speccy` binary against scratch workspaces) and the in-module
`#[cfg(test)]` suite at
`speccy-core/src/parse/vet_xml/serialize.rs` (the renderer and the
invocation/round planner). "red" is the pre-implementation state
(neither the VET serializer nor the vet routing existed, so the
fixtures had no command path / function to call); "green" is the
committed test passing under `cargo test --workspace`.

## Session 2026-06-10-T004-option-b — parser as single authority (DEC-008)

The vet append path was refactored so the `vet_xml` parser is the
single authority for both append derivation and body inertness,
removing the parallel tolerant text scan (`scan_existing` /
`block_body_ranges`) and the body-markup guard (`first_nested_vet_element`)
that re-implemented the parser's tag/heading/body-range definitions and
diverged from it across seven review rounds. SPEC-0055 gained DEC-008
(refines DEC-005; no block-grammar change). The parser exposes a new
`parse_in_flight` mode that relaxes only the last section's terminal-gate
rule; `plan_append` now derives from the typed `VetDoc`; and
`append_vet_under_lock` re-parses the would-be-new file through
`parse_in_flight` before any write.

The observable contract of REQ-004 is unchanged — every prior
integration test passes verbatim — so the proof here is the
behaviour-preserving invariants, not a new red/green command path.

### Invariant 1 — open trailing section parses in-flight but not strictly

<green>
`cargo test -p speccy-core --lib vet_xml`
test parse::vet_xml::tests::in_flight_accepts_open_trailing_section ... ok
test parse::vet_xml::tests::in_flight_matches_strict_on_a_complete_file ... ok
test parse::vet_xml::tests::in_flight_rejects_open_non_last_section ... ok
Asserts strict `parse` rejects an un-gated last section while
`parse_in_flight` accepts it, that a complete (fully gated) file parses
identically under both, and that an un-gated *non-last* section is still
rejected in-flight.
</green>

### Invariant 2 — a body that would corrupt the file is refused at write time

<red>
With the write-time `parse_in_flight` round-trip removed, a body whose
own line is a vet tag (open or close, including the Unicode-whitespace
and whitespace-padded variants) or a body-embedded `## Invocation`
heading lands a structurally-malformed VET.md silently — the exact
class the seven review rounds kept reopening.
</red>
<green>
`cargo test -p speccy-cli --test journal_append_vet`
- close_tag_in_body_is_rejected_at_write_time ... ok
- form_feed_separated_open_tag_in_body_is_rejected_at_write_time ... ok
- whitespace_padded_close_tag_in_body_is_rejected_at_write_time ... ok
- invocation_heading_in_body_does_not_skew_append_derivation ... ok
- any_accepted_body_produces_a_strict_parseable_vet ... ok
- gate_round_trip_accepts_legitimate_inline_tag_prose ... ok
Each drives the real `speccy journal append` binary: a body smuggling a
structural line is refused with VET.md absent, and a body that merely
mentions a tag mid-sentence (or quotes a `## Invocation` line) is
accepted and the gate-terminated file strict-parses. The unit twin
`parse::vet_xml::serialize::tests::rendered_block_round_trips_through_the_in_flight_parser`
pins the same invariant at the serializer layer.
</green>

CHK-006 (`drift_review_holistic_fix_gate_pass_resolves_past_vet`) and
CHK-007 (`simplifier_scan_after_gate_opens_invocation_two`) are
unchanged and still green; `speccy next` still emits `"kind":"ship"`
through the real freshness check.
