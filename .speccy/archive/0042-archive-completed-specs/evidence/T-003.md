`speccy archive --json` receipt output shape).

`demonstrated` because they are exercised by the new
`speccy-cli/tests/archive_json.rs` test binary, which parses

old hand-rolled writer in `main.rs` did not produce in a
serde-driven shape. The post-edit "green" form is the
passing test suite below.

## CHK roll call (REQ-009)

- CHK-023 — demonstrated. Test
  `archive_json_with_reason_emits_full_receipt` parses
  stdout JSON and asserts `schema_version == 1`,
  `archived.id == "SPEC-0001"`,
  `archived.archived_reason == "ship cleanup"`,
  `archived.to == ".speccy/archive/0001-artifact-parsers"`,
  `archived.from == ".speccy/specs/0001-artifact-parsers"`,
  `warnings == []`, and `archived_at` is a 10-char
  `YYYY-MM-DD` string.

- CHK-024 — demonstrated. Test
  `archive_json_on_in_progress_without_force_fails_with_empty_stdout`
  asserts that `speccy archive SPEC-0042 --json` on a spec
  with `status: in-progress` exits non-zero, emits zero
  bytes to stdout, and writes a stderr message containing
  the substring `in-progress`.

## Supplemental tests (not separate CHKs)

- `archive_json_without_reason_emits_null_reason` — asserts
  that omitting `--reason` produces the literal JSON
  `"archived_reason":null` substring and that the parsed
  `archived_reason` is `Value::Null` (not omitted, not the
  string `"null"`). Backs the task body's "absent values
  emit JSON `null`" contract.
- `archive_json_paths_use_forward_slashes` — asserts that
  `archived.from` and `archived.to` contain no `\` even on
  Windows hosts (where `Utf8Path` preserves native
  backslashes). Backs the task body's Windows-runner
  forward-slash contract.
