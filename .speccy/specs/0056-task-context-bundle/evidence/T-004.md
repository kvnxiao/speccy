# Evidence for SPEC-0056 T-004

Red-then-green paper trail for extending the `speccy context` bundle
with the inlined per-task journal (REQ-004). When the journal file
exists, the envelope inlines the frontmatter plus every block across
all rounds in file order; when it is absent, the envelope carries an
explicit `exists: false` marker with zero blocks and emission still
succeeds. The per-block JSON shape reuses SPEC-0055's `JsonJournalBlock`
and `to_json_journal_block` mapping (made `pub` this task) so `context`
and `journal show` cannot drift. Tests live in
`speccy-cli/tests/context.rs`. "Red" is established by the absence of
the `journal` field on the `ContextBundle` envelope before this task —
T-003 left the envelope with `schema_version`, `spec`, `intent`,
`task`, and `requirements` only, so the assertions below were
unreachable (the JSON carried no `journal` key, and the new fixtures
`journal_two_rounds` / `write_journal` did not exist).

## Session 2026-06-10-T004-attempt-1

### Scenario 1 — Full journal inlined: all eight blocks with matching rounds (CHK-006)

<red>
At HEAD before this task, `ContextBundle` had no `journal` field.
`serde_json::Value::get("journal")` returned `None`, so
`bundle_inlines_full_journal_with_all_blocks_and_rounds` panicked at
`.expect("bundle has journal section")`. There was no journal
projection at all, and `to_json_journal_block` was private to
`journal_show_output.rs`, so no shared mapping was reachable from
`context`.
</red>

<green>
`bundle_inlines_full_journal_with_all_blocks_and_rounds` writes a
fixture journal (`journal_two_rounds`) with rounds 1–2 laid out in a
round-monotonic file order the `journal_xml` parser accepts: round 1
carries one `<implementer>`, three `<review>` (business pass, tests
pass, security blocking), and one `<blockers>`; round 2 carries one
`<implementer>` and two `<review>` (security pass, style pass) — eight
blocks, 2 implementer + 5 review + 1 blockers. The test asserts:

- `journal.exists == true`.
- `journal.blocks` has exactly 8 entries.
- block kinds count to 2 implementer / 5 review / 1 blockers.
- the `round` attributes in file order are `[1,1,1,1,1,2,2,2]`,
  proving file-order preservation and correct round projection.
- distinctive body markers (`IMPL_R1_MARKER`,
  `REVIEW_R1_SECURITY_MARKER`, `BLOCKERS_R1_MARKER`, `IMPL_R2_MARKER`,
  `REVIEW_R2_STYLE_MARKER`) appear in the payload — retry context
  (prior handoffs, verdicts, blockers directives) is all present.
- the one `blocking`-verdict block carries `persona == "security"`,
  proving review attributes ride through the shared mapping.

The projection routes every entry through
`crate::journal_show_output::to_json_journal_block` (made `pub` this
task) so `context` and `journal show` share one journal-to-JSON
mapping (REQ-004 / DEC-002). Test command:
`cargo test -p speccy-cli --test context`.
</green>

### Scenario 2 — Absent journal yields explicit empty marker and exit 0 (CHK-007)

<red>
Before this task, with no `journal` field on the envelope,
`bundle_journal_absent_yields_explicit_empty_marker_and_success`
panicked at `.expect("bundle has journal section")`. There was no
empty-journal marker and no defined behaviour for an absent file in
the bundle.
</red>

<green>
`bundle_journal_absent_yields_explicit_empty_marker_and_success`
builds a spec + task with NO journal file written and asserts the
emission succeeds (the `?` would propagate any error), `journal.exists
== false`, and `journal.blocks` is empty.
`binary_journal_absent_exits_zero` runs the real `speccy` binary
(`assert_cmd`) and asserts `.success()` (process exit code 0) with
stdout containing `"exists":false` — proving the exit-0 contract at the
process boundary, not just the absence of a Rust error. A round-1
implementer legitimately has no journal yet (DEC-004), so absence is
normal rather than anomalous (in contrast to `journal show`, whose
call sites treat absence as an anomaly).

```
$ speccy context SPEC-0042/T-001 --json
{"schema_version":1,...,"journal":{"exists":false,"blocks":[]}}
(exit 0)
```
</green>
