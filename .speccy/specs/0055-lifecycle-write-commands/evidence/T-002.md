# Evidence — SPEC-0055 / T-002

`speccy journal append` for per-task journals with CLI-stamped attributes
and an advisory file lock (REQ-003, REQ-005).

## How to reproduce

```
cargo test -p speccy-core journal_xml::serialize
cargo test -p speccy-cli --test journal_append
```

Each `<red>` transcript below was captured by temporarily disabling the
production capability under test (a one-line mutation, noted inline),
running the test, then restoring the code. The `<green>` transcript is
the same test against the restored (shipped) code.

## Scenario evidence

### Scenario 1 — fresh `implementer` append creates journal with frontmatter and round=1 (CHK-004)

Test: `fresh_implementer_creates_journal_with_frontmatter_and_round_one`
(`speccy-cli/tests/journal_append.rs`).

<red>
Mutation: `derive_round` opener changed from `map_or(1, …)` to
`map_or(0, …)` in `speccy-core/src/parse/journal_xml/serialize.rs`, so a
round-opening `implementer` on a fresh file derives `round=0` instead of
`round=1`. The CLI's pre-write round-trip parse then rejects the block —
the test returns the parse error rather than observing `round=1`.

```
$ cargo test -p speccy-cli --test journal_append \
    fresh_implementer_creates_journal_with_frontmatter_and_round_one -- --exact
running 1 test
test fresh_implementer_creates_journal_with_frontmatter_and_round_one ... FAILED

---- fresh_implementer_creates_journal_with_frontmatter_and_round_one stdout ----
Error: InvalidJournalAttribute { path: ".../journal/T-001.md", element: "implementer", attribute: "round", value: "0", reason: "round must be a positive integer (regex `[1-9][0-9]*`)", offset: 72 }

failures:
    fresh_implementer_creates_journal_with_frontmatter_and_round_one
test result: FAILED. 0 passed; 1 failed; 0 ignored; 0 measured; 6 filtered out; finished in 0.40s
error: test failed, to rerun pass `-p speccy-cli --test journal_append`
```

The fresh-create path cannot produce a round-1-first block when the
opener is wrong; the failure is the parser refusing the non-positive
round.
</red>

<green>
Restored code: a round-opening `implementer` on a fresh file derives
`round=1`, the file is created with CLI-stamped frontmatter, and the
post-append parse accepts it.

```
$ cargo test -p speccy-cli --test journal_append \
    fresh_implementer_creates_journal_with_frontmatter_and_round_one -- --exact
   Compiling speccy-core v1.0.0 (/Users/kevin/src/speccy/speccy-core)
   Compiling speccy-cli v1.0.0 (/Users/kevin/src/speccy/speccy-cli)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 2.39s
     Running tests/journal_append.rs (target/debug/deps/journal_append-400271fa68991356)
running 1 test
test fresh_implementer_creates_journal_with_frontmatter_and_round_one ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 6 filtered out; finished in 0.42s
```

Before the command the file does not exist (`!jpath.exists()`); after
`speccy journal append SPEC-0042/T-001 --block implementer --model
test-model` with a body on stdin the file parses under
`parse_journal_xml`, frontmatter carries `spec=SPEC-0042`, `task=T-001`,
a non-empty CLI-stamped `generated_at`, and the single `<implementer>`
block has `round=1` and an ISO8601 `Z` `date` the caller never supplied
(there is no `--date` flag).
</green>

### Scenario 2 — unknown-persona `review` exits non-zero, journal byte-identical (CHK-005)

Test: `unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged`
(`speccy-cli/tests/journal_append.rs`).

<red>
Mutation: the persona-registry check in `validate_and_render_block`
(`speccy-core/src/parse/journal_xml/serialize.rs`) was short-circuited
(`if false && !PERSONAS_ALL.contains(&p)`), so an unknown `--persona`
passes validation, the append succeeds, and the bytes change.

```
$ cargo test -p speccy-cli --test journal_append \
    unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged -- --exact
running 1 test
test unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged ... FAILED

---- unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged stdout ----
Unexpected success
command=`... "speccy" "journal" "append" "SPEC-0042/T-001" "--block" "review" "--model" "test-model" "--persona" "not-a-persona" "--verdict" "pass"`
stdin=`"a review body"`
code=0
stdout=""
stderr=""

failures:
    unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged
test result: FAILED. 0 passed; 1 failed; 0 ignored; 0 measured; 6 filtered out; finished in 0.42s
```

With the registry check disabled the append exits `code=0` and writes —
the test's expected non-zero exit and byte-equality both fail.
</red>

<green>
Restored code: the `not-a-persona` value is rejected before any write.

```
$ cargo test -p speccy-cli --test journal_append \
    unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged -- --exact
running 1 test
test unknown_persona_review_exits_nonzero_and_leaves_bytes_unchanged ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 6 filtered out; finished in 0.48s
```

A seed `implementer` block is appended first so a round exists, and the
journal bytes are captured. The `review --persona not-a-persona` append
exits non-zero with `invalid persona` on stderr, and the captured bytes
are unchanged (validation runs before any write).
</green>

### Scenario 3 — eight concurrent `review` appends serialize cleanly (CHK-008)

Test: `eight_concurrent_review_appends_serialize_cleanly`
(`speccy-cli/tests/journal_append.rs`).

<red>
Mutation: `LockGuard::acquire` (`speccy-cli/src/journal.rs`) returns the
guard immediately without taking the advisory `flock`, so the eight
concurrent appends race on read-modify-write and lose writes.

```
$ cargo test -p speccy-cli --test journal_append \
    eight_concurrent_review_appends_serialize_cleanly -- --exact
running 1 test
test eight_concurrent_review_appends_serialize_cleanly ... FAILED

failures:
thread 'eight_concurrent_review_appends_serialize_cleanly' panicked at speccy-cli/tests/journal_append.rs:298:5:
assertion `left == right` failed: all eight review blocks must be present
  left: 1
 right: 8

failures:
    eight_concurrent_review_appends_serialize_cleanly
test result: FAILED. 0 passed; 1 failed; 0 ignored; 0 measured; 6 filtered out; finished in 0.45s
```

Without the lock, seven of the eight writes are lost (only 1 `<review>`
block survives instead of 8).
</red>

<green>
Restored code: the advisory lock serializes the eight appends; all eight
blocks land and the file parses.

```
$ cargo test -p speccy-cli --test journal_append \
    eight_concurrent_review_appends_serialize_cleanly -- --exact
running 1 test
test eight_concurrent_review_appends_serialize_cleanly ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 6 filtered out; finished in 0.61s
```

After a seed `implementer` block, eight threads each spawn a
`speccy journal append --block review` child against the same journal.
All eight exit success; the resulting file parses under
`parse_journal_xml`, and exactly eight `<review>` blocks are present with
no interleaving.
</green>

## Other tested done-when (not `demonstrated` CHKs)

These back REQ-003 / REQ-005 done-when clauses that are not roll-called
as `demonstrated` CHKs; their green proof is the workspace test gate
below.

- `two_concurrent_round_opening_appends_get_distinct_rounds` — two
  concurrent round-opening `implementer` appends derive distinct rounds
  `{1, 2}` because derive→validate→append runs under the per-file lock.
- `held_lock_makes_waiting_append_time_out_nonzero` — with the sidecar
  lock held by the test process, a waiting append exits non-zero after
  roughly the 10s interval (asserted `>= 9s`), names the journal path on
  stderr, and leaves the journal byte-identical.
- `review_with_no_round_exits_nonzero_and_journal_stays_absent`,
  `empty_body_exits_nonzero` — each exits non-zero with the journal still
  absent.
- `speccy-core/src/parse/journal_xml/serialize.rs` `#[cfg(test)]`: round
  derivation (fresh→1, attaching-on-fresh rejected, round-2 after
  round-1, review attaches to current round), validation (missing
  persona/verdict, unknown persona/verdict, empty body, nested journal
  markup, prose `reviewers` not mistaken for markup), and round-trip
  parse of rendered output.
