all four authoring callsites (T-003..T-006) have landed. T-007 edits no source

## How the red phase is captured

These are cross-set assertions: they can only pass once T-003..T-006 have wired
the branch-guard include and narrow-staging commands into the four authoring
callsites. The `include_dir`-embedded `RESOURCES` bundle compiles against the
live `resources/` tree, so the meaningful red baseline is the pre-callsite
source state — the tree at the SPEC-0059 T-002 boundary
(`57c0a89`), after the shared modules existed (T-002) but before any authoring
skill `{% include %}`'d the branch-guard or gained a narrow-staging command.
Against that source the scoping and staging assertions fail; against HEAD they
pass.

The red counts below were captured against the real `57c0a89` tree via
`git show <commit>:<path>`; the green counts are the live HEAD sources the
`RESOURCES` bundle embeds.

### Scenario 2 — Narrow staging, no clean-tree gate (CHK-012; REQ-009)

<green>
Post-callsite (HEAD): each authoring body carries its narrow `git add <path>`

Note on the negative assertion: this scenario does **not** bare-substring the
forbidden `git add -A` / `git add .` forms, because the authoring bodies
legitimately *forbid* those forms by name in prohibition prose (a substring
check would false-match the prohibition — the same trap the sibling plan,
decompose, and amend staging tests already document). It anchors on the
positive narrow-staging command and on the absence of the stable `git stash`
token, which the SPEC's "no stashing or refusal on a dirty tree" non-goal makes
the right marker for "no clean-tree gate."

