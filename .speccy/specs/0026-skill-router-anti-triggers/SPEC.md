# SPEC-0026: Anti-triggers in skill descriptions to reduce host-router mis-fires

## Summary

The eight shipped `speccy-*` SKILL.md descriptions are affirmative-only
today. Each lists trigger phrases like "Use when the user says
'implement T-003'". None names workspace preconditions, none routes
to alternative skills when those preconditions are missing, and none
discourages firing on common mis-routes. As a result, the host's
skill router (Claude Code or Codex) can fire `speccy-work` against
a workspace with no `TASKS.md`, or `speccy-ship` while tasks are
still in flight, costing the user a wasted turn each time.

This spec adds a uniform tail structure to every shipped `speccy-*`
description: a `Requires:` clause naming workspace preconditions
(or `Requires: no preconditions` for `speccy-init` and
`speccy-brainstorm`), optional `If <state-missing> → prefer
<other-skill>` routing cues scaled to the skill's actual
preconditions, and one `Do NOT trigger…` line per skill targeting
the most realistic mis-route. The change is content-only — 16 source
files edited (8 skills × 2 host trees of `.tmpl`), 16 dogfood mirrors
regenerated by `speccy init --force`. The Rust CLI, artifact
grammars, prompt templates, and renderer pipeline are all
unchanged.

The intervention point is deliberate. F-2 in `.speccy/BACKLOG.md`
identifies the host's skill router as the leverage point for
reducing wasted turns. The router's only signal at decision time is
the description text in SKILL.md frontmatter; SKILL.md bodies,
runtime guards, and host-side hooks are too late. The change must
land at the description layer or it does not address the stated
problem. Anti-triggers are not a documented Anthropic best practice
but appear in shipped Anthropic skills (e.g., `docx` ships with
"Do NOT use for PDFs, spreadsheets, Google Docs…"); the chosen
shape (preconditions + routing + one Do-NOT line) aligns more
closely with the "affirmative-specific" guidance the official docs
emphasize than pure negation does.

Per Principle 1 ("feedback, not enforcement") and Principle 2
("deterministic core, intelligent edges"), the speccy CLI does not
validate description structure. A `speccy check` lint that asserted
"every skill has a Requires line" was explicitly considered and
rejected — speccy is agent-agnostic and must not enforce
agent-harness skill shape. Future skill #9 not following the pattern
is a code-review issue, not a CLI issue.

## Goals

<goals>
- Every shipped `speccy-*` SKILL.md description carries a uniform
  tail structure: a `Requires:` clause naming workspace
  preconditions, optional `If <state-missing> → prefer
  <other-skill>` routing cues for skills with hard preconditions,
  and one `Do NOT trigger…` line naming the most realistic
  mis-route.
- Source-of-truth lives in the two `.tmpl` files per skill at
  `resources/agents/.claude/skills/<name>/SKILL.md.tmpl` and
  `resources/agents/.agents/skills/<name>/SKILL.md.tmpl`. Dogfood
  mirrors at `.claude/skills/<name>/SKILL.md` and
  `.agents/skills/<name>/SKILL.md` regenerate from those sources
  via `speccy init --force --host claude-code` and
  `speccy init --force --host codex`.
- Every description stays within 1024 characters total length
  (Codex's code-enforced hard-reject cap; the binding constraint
  because Claude Code's documented 1536-char cap merely truncates
  rather than rejects).
- The affirmative trigger phrases inherited from today's
  descriptions are preserved where length allows. If a skill's
  revised description would exceed 1024 chars, the affirmative
  phrase list is compressed first; the `Requires:` / routing / Do-NOT
  clauses are never dropped to fit the cap.
</goals>

## Non-goals

<non-goals>
- No `speccy check` rule, lint, or CI structural validation that
  the descriptions follow the prescribed tail shape. Speccy's CLI
  is agent-agnostic (Principle 2: deterministic core, intelligent
  edges); enforcing agent-harness skill description structure
  is host-side concern. Future skill #9 not following the pattern
  is caught by code review, not CLI machinery.
- No runtime guard in SKILL.md bodies that the agent reads after
  invocation to verify workspace state. The router-time decision
  is what F-2 targets; body-level guards are F-6 hook-templates
  territory and out of scope here.
- No new renderer machinery, no shared single frontmatter source
  per skill, no per-host overlay, no `description_tail` template
  variable. The two `.tmpl` files per skill stay byte-identical for
  the frontmatter portion by convention; git-diff catches drift
  across the two files on PR review.
- No measurement infrastructure for "did this reduce wasted
  firings". Host-router decision-making is a blackbox; the metric
  is not observable from outside the host. This SPEC ships with
  that limitation documented in `<assumptions>`.
- No semantic change to any existing affirmative trigger phrase
  list. The "Use when the user says X, Y, Z" lists stay in place
  except when the total revised description would exceed 1024
  chars, in which case the affirmative phrase list (not the new
  tail clauses) is compressed.
- No new shipped skill. The set of eight
  (`init`, `brainstorm`, `plan`, `tasks`, `work`, `review`,
  `amend`, `ship`) is unchanged. Only the `description:` field of
  each existing skill's frontmatter changes.
- No change to any SKILL.md body, any prompt template under
  `resources/modules/prompts/`, or any rendered prompt the Rust
  CLI emits. F-2 is frontmatter-only.
- No PSA, documentation page, or `.speccy/ARCHITECTURE.md` update
  describing the tail-clause convention. The convention is
  demonstrated by the eight in-tree examples; documenting it
  separately would invite drift between the doc and the examples.
</non-goals>

## User Stories

<user-stories>
- As a solo developer chatting with Claude Code in a workspace
  with no `TASKS.md`, when I say "fix this bug" or "refactor that
  function", I want the router to *not* fire `speccy-work` (which
  has no task scope to act on), so I am not interrupted by a
  misrouted skill walkthrough and a wasted turn.
- As an AI agent picking a skill from descriptions, I want each
  description to name the workspace state the skill expects, so I
  can route to the right skill (e.g., `speccy-tasks` instead of
  `speccy-work`) when the prerequisite state is missing.
- As a maintainer adding a future skill to the speccy pack, I want
  the existing eight descriptions to demonstrate the tail-clause
  pattern visibly, so I follow it by copy-shape without needing a
  separate lint, ADR, or doc to know the convention exists.
- As a developer reviewing a PR that edits one skill's frontmatter,
  I want the two `.tmpl` files (Claude Code + Codex) to be obvious
  byte-mirrors of each other, so a drifted edit on one host fails
  the eye-test on `git diff` review without a tool.
</user-stories>

## Design

### Approach

This SPEC ships in five mechanical steps. None requires renderer
or CLI code changes; the entire SPEC is content-only.

1. **Edit the 16 source .tmpl files.** For each of the eight
   speccy-* skills, edit both
   `resources/agents/.claude/skills/<name>/SKILL.md.tmpl` and
   `resources/agents/.agents/skills/<name>/SKILL.md.tmpl` so the
   frontmatter `description:` field carries the revised text per
   the per-skill matrix below. The two .tmpl files for the same
   skill must remain byte-identical for the frontmatter portion
   (the include directives in the body are already host-specific
   but identical-by-convention today).

2. **Regenerate the 16 dogfood mirrors.** Run
   `cargo run -- init --force --host claude-code` and
   `cargo run -- init --force --host codex` in this workspace.
   The renderer expands `{% include "modules/skills/<name>.md" %}`
   against the new frontmatter, producing updated dogfood files
   at `.claude/skills/<name>/SKILL.md` and
   `.agents/skills/<name>/SKILL.md`.

3. **Verify char counts.** Run a one-off check (a temporary
   test or a shell pipe over the frontmatters) confirming every
   description stays ≤ 1024 Unicode chars. If any exceeds, compress
   the affirmative trigger phrase list (not the new tail clauses)
   until it fits.

4. **Sanity-check rendering.** Read the rendered dogfood files end
   to end and confirm: (a) the tail clauses survived rendering;
   (b) the body content (which lives in `resources/modules/skills/`)
   is unchanged; (c) `git diff` against this workspace's committed
   dogfood files shows only the description deltas.

5. **Run the hygiene gate.** `cargo test --workspace`,
   `cargo clippy --workspace --all-targets --all-features -- -D
   warnings`, `cargo +nightly fmt --all --check`, `cargo deny
   check`. None of the four should flag this SPEC's changes
   because no Rust code is touched; this is a safety check.

#### Per-skill precondition matrix

The matrix below names each skill's `Requires:` clause, routing
cues, and Do-NOT mis-route target. Exact wording is implementer
discretion; the matrix is the design contract.

| Skill | Requires | Routing cues | Do-NOT (mis-route discouraged) |
|---|---|---|---|
| `speccy-init` | no preconditions | — | when `.speccy/` already exists (use `speccy-amend` or `speccy-plan`) |
| `speccy-brainstorm` | no preconditions | — | when the user has named a sharp slice with clear scope (go directly to `speccy-plan`) |
| `speccy-plan` | `.speccy/` + `AGENTS.md` | → `speccy-init` | on fuzzy asks lacking concrete scope (prefer `speccy-brainstorm` first) |
| `speccy-tasks` | `SPEC.md` exists | → `speccy-plan` | as part of a SPEC amendment (`speccy-amend` reconciles `TASKS.md`) |
| `speccy-work` | `TASKS.md` with ≥1 pending task | → `speccy-tasks`, `speccy-plan`, `speccy-init` | on generic "fix bug" / "refactor X" asks unscoped to a Speccy task |
| `speccy-review` | a task in `in-review` state | → `speccy-work` (if no in-review task), `speccy-ship` (if all complete) | on generic "review this PR / code" asks unrelated to Speccy task-state review |
| `speccy-amend` | an existing `SPEC.md` | → `speccy-plan` (if no SPEC yet) | for cosmetic edits to `SPEC.md` that do not change Requirements |
| `speccy-ship` | all tasks `completed` | → `speccy-work` | while any task is still `pending` or `in-progress` |

The matrix is the load-bearing contract for the description edits.
A future skill addition (skill #9) adds a row here.

#### Brainstorm-locked speccy-brainstorm description

The `speccy-brainstorm` description was the only one where the
chosen shape diverged from the uniform template (it omits routing
cues entirely). The brainstorm conversation that produced this
SPEC selected the following text as the authoritative reference;
implementer wording may vary within reason but must preserve the
shape:

```text
Atomize a fuzzy ask into first-principle requirements before any
SPEC.md is written. Walks the user through a Socratic exchange (one
question at a time, 2-3 alternative framings, silent assumptions,
open questions) and stops at a hard gate until the user explicitly
approves the framing. Use when the user has a fuzzy idea, says
"help me think about", "brainstorm with me", "I want to spec out X
but I'm not sure where to start", or before invoking speccy-plan on
an unclear ask. Requires: no preconditions. Do NOT trigger when the
user has named the slice and the scope is clear — go straight to
speccy-plan.
```

### Decisions

<decision id="DEC-001" status="accepted">
#### DEC-001: 1024-char Codex cap is the binding length constraint

**Status:** Accepted

**Context:** Anthropic documents a 1536-char cap on `description`
+ `when_to_use` combined for Claude Code skills, applied as
truncation. OpenAI's Codex CLI has an undocumented 1024-char hard
cap on `description` alone, enforced in
`codex-rs/core-skills/src/loader.rs:MAX_DESCRIPTION_LEN`; a
description exceeding it causes the skill to be rejected at load
time. Filed as
[openai/codex#13941](https://github.com/openai/codex/issues/13941).
This is the tighter constraint; Codex rejects while Claude Code
merely truncates.

**Decision:** Every shipped speccy-* description stays ≤ 1024
Unicode characters total. The cap is documented in the
`<assumptions>` block of this SPEC and noted in REQ-005's body so
future contributors do not bump descriptions past Codex's cap
when adding a ninth skill.

**Alternatives:** (a) Target 1536 (Claude Code's cap), accept
that Codex would silently reject some descriptions. Rejected: the
loop must work on both hosts. (b) Maintain divergent per-host
descriptions, target each host's cap independently. Rejected:
doubles the maintenance surface and makes drift more likely; the
2 .tmpl files per skill stay byte-identical for the description
field today and this SPEC preserves that.

**Consequences:** Current descriptions range 232-540 chars, so
the cap leaves 484-792 chars of headroom for the tail clauses.
The largest current description is `speccy-review` at ~538 chars;
the tail additions there must be the most compact across the
eight skills.
</decision>

<decision id="DEC-002" status="accepted">
#### DEC-002: No CLI lint enforces description structure

**Status:** Accepted

**Context:** A `speccy check` rule that asserted "every
speccy-* SKILL.md description contains `Requires:` and at least
one routing cue or `Do NOT trigger` line" was considered. It
would catch future skill #9 not following the convention.

**Decision:** No such lint is added. Speccy's CLI is
agent-agnostic per Principle 2 ("deterministic core, intelligent
edges"); enforcing agent-harness skill description structure
crosses the boundary. The CLI should not know what shape a skill
description takes.

**Alternatives:** (a) Add `speccy check` lint for description
shape. Rejected: violates agent-agnostic principle. (b) Add a
unit test in `speccy-cli/tests/skill_packs.rs` that asserts every
shipped skill has the tail clauses. Considered: test-side
enforcement is less load-bearing than a CLI lint because the test
catches drift only on the shipped resource bundle, not on user
projects, which is the right scope. Could be added in a follow-up
SPEC if dogfooding shows drift on skill #9 onward; out of scope
here.

**Consequences:** Future skill additions rely on code-review
discipline and the in-tree example pattern to follow the
tail-clause convention. If dogfooding shows this isn't enough,
a follow-up SPEC can add an opt-in unit-test check (not a CLI
lint).
</decision>

<decision id="DEC-003" status="accepted">
#### DEC-003: Two byte-identical .tmpl files per skill — not a shared single source

**Status:** Accepted

**Context:** Today each skill has two `.tmpl` files (one per host
tree) whose YAML frontmatter is byte-identical by convention. A
shared single frontmatter source per skill (composed into both
host trees by an extended renderer) would eliminate drift class
entirely. Eight new source files, plus renderer changes to
support per-host overlay for future divergence.

**Decision:** Keep the two `.tmpl` files per skill, both
hand-edited, kept in sync by git-diff review on PR. No shared
single source. No renderer change.

**Alternatives:** Promote frontmatter to a single shared source
per skill. Rejected for v1: introduces renderer surface (per-host
overlay) we do not need yet; today's two `.tmpl` files are
byte-identical for the description field and a coordinated edit
covers both. Revisit when Claude-Code-specific frontmatter fields
(`allowed-tools`, etc.) enter the picture, since at that point
the two files would no longer be byte-identical and a true
divergence mechanism becomes necessary.

**Consequences:** Editing one skill's description requires
touching two .tmpl files per skill (16 edits total this SPEC).
Reviewers eye-check the .tmpl diffs and confirm symmetry. If a
future contributor edits only one of the pair, the diff makes
that obvious.
</decision>

<decision id="DEC-004" status="accepted">
#### DEC-004: Anti-triggers live in `description`, not in SKILL.md body

**Status:** Accepted

**Context:** Two ways to discourage misrouted firings: (a) put
the anti-trigger in the description so the host's skill router
sees it at decision time; (b) put a body-level preconditions
guard the agent reads after the router has already fired.

**Decision:** Anti-triggers go in the description. F-2's stated
goal is "reduce wasted firings" — i.e., reduce router decisions,
not reduce post-fire walkthrough cost. The only signal the router
sees is the description; body-level guards do not help at the
decision point. Body-level guards are a separate feature (closer
to F-6's hook-template proposal) and out of scope here.

**Alternatives:** Body-level `## Preconditions` section the
agent reads after invocation. Rejected: addresses a different
problem (recovering from a bad firing, not preventing it). May
land later as a complementary feature.

**Consequences:** This SPEC is description-only. If a future
SPEC adds body-level preconditions, the two layers are
complementary (description = pre-firing prevention; body =
post-firing recovery).
</decision>

<decision id="DEC-005" status="accepted">
#### DEC-005: All eight skills get a `Do NOT trigger…` line — no exemptions

**Status:** Accepted

**Context:** Walked through every shipped skill looking for one
where no realistic mis-route exists. None found. Even
`speccy-init` (which has no workspace preconditions) has a
real mis-route — being invoked on a workspace where `.speccy/`
already exists. Even `speccy-brainstorm` (an optional-by-design
skill) can mis-trigger when the user has already named a sharp
slice.

**Decision:** All eight skills carry exactly one `Do NOT trigger`
clause. Uniformity is a feature: the convention is visible by
example across the whole skill pack, so skill #9 inherits it by
shape-matching.

**Alternatives:** Allow skills with no obvious mis-route to omit
the Do-NOT line. Rejected: every speccy-* skill has at least one
realistic mis-route; uniformity is cheaper than per-skill
discretion at this scale.

**Consequences:** Eight Do-NOT lines to write. Each is one
sentence. The total char cost is ~80-120 chars per skill,
well within the 484-792-char headroom under the 1024 cap.
</decision>

  the affirmative trigger phrases, the new tail clauses may be

## Notes

The brainstorm phase before this SPEC considered three
description shapes and three source-of-truth strategies. The
rejected alternatives are kept here for posterity; promoting any
of them to a `<decision>` block was not warranted because each
rejection rests on a Speccy principle already documented elsewhere.

**Rejected description shapes:**

- *Pure negative phrase patterns* ("Do NOT trigger when the user
  says X"). Closer to Anthropic's shipped `docx` example, but
  the official Skills docs emphasize affirmative specificity over
  negation. The chosen shape (`Requires:` + routing + one
  Do-NOT line) leans more affirmative-specific.
- *Splitting affirmative triggers and anti-triggers across
  `description` and `when_to_use` fields*. Anthropic's frontmatter
  supports a separate `when_to_use` field, but Codex does not
  (Codex caps the `description` field alone). Splitting would
  produce divergent host wiring; the chosen shape uses
  `description` only and remains compatible with both hosts.

**Rejected source-of-truth strategies:**

- *Shared single frontmatter source per skill with renderer
  overlay* (DEC-003 captures this). Eliminates drift class but
  introduces renderer surface we do not need yet.
- *Hand-edit all 32 files (16 source + 16 dogfood) directly, plus
  a `speccy check` lint to assert the four host-tree frontmatters
  match per skill*. Rejected on the principle that the CLI should
  not know about skill description structure (DEC-002).

**Anti-triggers in the official skill ecosystem:**

Anthropic does not publish guidance recommending anti-triggers,
but at least one shipped Anthropic skill (`docx`) uses them:
"Do NOT use for PDFs, spreadsheets, Google Docs, or general
coding tasks unrelated to document generation". Other shipped
Anthropic skills (e.g., `pptx`) omit anti-triggers entirely.
Practice is mixed in the wild; Speccy's choice is "every speccy-*
description gets one Do-NOT line, uniformly" because the in-tree
example pattern is the cheapest way to teach the convention to
future skill authors.

**Why a description-only change is enough for F-2:**

The host's skill router decides which skill to fire based on the
SKILL.md frontmatter, not the body. SKILL.md body content, runtime
preconditions, and host-side hooks are downstream of the routing
decision. F-2's goal is to reduce wasted firings, so the
description is the only intervention point that addresses it
without changing the host's behavior. Other proposed levers
(body-level preconditions in F-6, host-side hook templates) are
complementary but address different problems.
